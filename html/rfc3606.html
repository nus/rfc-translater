1. The Internet-Standard Management Framework
1.インターネット標準管理フレームワーク

For a detailed overview of the documents that describe the current Internet-Standard Management Framework, please refer to section 7 of RFC 3410 [RFC3410].

現在のインターネット標準の管理フレームワークを記述したドキュメントの詳細な概要については、RFC 3410 [RFC3410]のセクション7を参照してください。

Managed objects are accessed via a virtual information store, termed the Management Information Base or MIB. MIB objects are generally accessed through the Simple Network Management Protocol (SNMP). Objects in the MIB are defined using the mechanisms defined in the Structure of Management Information (SMI). This memo specifies a MIB module that is compliant to the SMIv2, which is described in STD 58, RFC 2578 [RFC2578], STD 58, RFC 2579 [RFC2579] and STD 58, RFC 2580 [RFC2580].

管理対象オブジェクトが仮想情報店を介してアクセスされ、管理情報ベースまたはMIBと呼ばれます。 MIBオブジェクトは、一般的に簡易ネットワーク管理プロトコル（SNMP）を介してアクセスされます。 MIBのオブジェクトは、管理情報（SMI）の構造で定義されたメカニズムを使用して定義されています。このメモは、STD 58、RFC 2578 [RFC2578]、STD 58、RFC 2579 [RFC2579]とSTD 58、RFC 2580 [RFC2580]に記載されているSMIv2のに準拠しているMIBモジュールを指定します。

2. Overview
2.概要

The purpose of this memo is to provide additional capabilities, not found in the ATM-MIB [RFC2515], which are needed to manage ATM interfaces. This memo addresses the following areas:

このメモの目的は、ATMインターフェイスを管理するために必要とされるATM-MIB [RFC2515]に見出されない追加機能を提供することです。このメモは、次の領域に対処します。

- ATM Switch Support - ATM Service Support - ATM Host Support

- ATMスイッチサポート - ATMサービスサポート - ATMホストのサポート

In addition, this memo also provides ATM trap support.

また、このメモはまた、ATMトラップサポートを提供します。

2.1. Background
2.1。バックグラウンド

In addition to the MIB module defined in this memo, other MIB modules are necessary to manage ATM interfaces, links and cross-connects. Examples include MIB II for general system and interface management ([RFC2863]), the DS3 ([RFC2496]) or SONET MIBs ([RFC3592]) for management of SONET and DS3 physical interfaces, and, as appropriate, MIB modules for applications that make use of ATM, such as SMDS [RFC1694] and LAN Emulation [ATM Forum LANE]. These MIB modules are outside the scope of this specification.

このメモで定義されたMIBモジュールに加えて、他のMIBモジュールは、ATMインターフェイス、リンクとクロスコネクトを管理するために必要です。例としては、MIB一般的なシステムとインターフェース管理（[RFC2863]）のためにII、SONETおよびDS3物理インターフェイスの管理のためのDS3（[RFC2496]）またはSONETのMIB（[RFC3592]）、および、必要に応じて、アプリケーションのためのMIBモジュールを含むことこのようSMDS [RFC1694]とLANエミュレーション[ATMフォーラムのLANE]としてATMを使用してください。これらのMIBモジュールは、本明細書の範囲外です。

This MIB module also requires the use of the ATM-MIB module defined in [RFC2515] and ATM-specific textual conventions defined in [RFC2514].

このMIBモジュールはまた、[RFC2515]及び[RFC2514]で定義されたATM固有テキストの表記法で定義されたATM-MIBモジュールの使用を必要とします。

ATM Endpoint applications such as ATM LAN Emulation or Classical IP-over-ATM Clients and Servers use ATM to establish SVC/PVC connections for exchanging control and data information. The agents of these ATM applications must provide the network manager with information on the SVC/PVCs in use and which applications are using them. The information can be made generic so as to apply to all ATM applications. This memo defines extensions to the ATM-MIB [RFC2515] in order to support this.

そのようなATM LANエミュレーションやクラシックIPオーバーATMクライアントとサーバーなどのATMエンドポイントのアプリケーションは、制御およびデータ情報を交換するためのSVC / PVC接続を確立するためにATMを使用しています。これらのATMアプリケーションのエージェントは、アプリケーションがそれらを使用しているSVC /使用中のPVCとの情報をネットワーク管理者に提供しなければなりません。すべてのATMアプリケーションに適用するような情報は、一般的な行うことができます。このメモは、これをサポートするためにATM-MIB [RFC2515]の拡張機能を定義します。

The current specification of this supplemental ATM2-MIB is based on SNMPv2 SMI.

この補足ATM2-MIBの現在の仕様は、SNMPv2 SMIに基づいています。

3.1.1. ATM SVC VP Cross-Connect Table
3.1.1。 ATM SVC VPクロスコネクト表

This table provides the SVC VP Cross-Connect (SVPC) information. The equivalent PVC VP Cross-Connect table is defined in [RFC2515]. This table also includes cross-connect information for Soft PVPs.

この表は、SVC VPクロスコネクト（SVPC）の情報を提供します。同等のPVC VPクロスコネクトテーブルは、[RFC2515]で定義されています。この表はまた、ソフトのPVPのためにクロスコネクト情報を含みます。

This table contains configuration and state information of all SVC VP point-to-point, point-to-multipoint, or multipoint-to-multipoint VP cross-connects.

このテーブルは、すべてのSVC VPポイントツーポイント、ポイントツーマルチポイント、またはマルチポイントツーマルチVPクロスコネクトの構成及び状態情報を含みます。

This table has read-only access and can be used to monitor the cross-connects which connect the VPLs together in an ATM switch or network. The atmSvcVpCrossConnectIndex is used to associate the related SVC VPLs that are cross-connected together. The atmSvcVpCrossConnectRowStatus object has read-write access to allow for tear-down.

このテーブルは、読み取り専用アクセスをしており、ATMスイッチまたはネットワークで一緒のVPLsを接続するクロスコネクトを監視するために使用することができます。 atmSvcVpCrossConnectIndexは一緒に相互接続される関連のSVCのVPLsを関連付けるために使用されます。 atmSvcVpCrossConnectRowStatusオブジェクトは、読み書きしたアクセスをティアダウンを可能にします。

The ATM SVC VP Cross-Connect Table models each bi-directional Switched Virtual Circuit (SVC) VP cross-connect as a set of entries in the atmSvcVpCrossConnectTable. A point-to-point VPC cross-connect is modeled as one entry; a point-to-multipoint (N leafs) VPC cross-connect as N entries in this table; and a multipoint-to-multipoint (N parties) VPC cross-connect as N(N-1)/2 entries in this table. In the latter cases, all the N (or N(N-1)/2) entries are associated with a single VPC cross-connect by having the same value of atmSvcVpCrossConnectIndex.

3.4. ATM-related Trap Support
3.4。 ATM関連のトラップのサポート

Traps are defined to detect changes in the status of permanent VPLs and VCLs. The current up/down status of each permanent VPL or VCL is indicated by the atmVplOperStatus or atmVclOperStatus object, respectively. Several tables and objects and one trap are defined in order to help network managers quickly and efficiently detect changes in the status of permanent virtual links. Through use of these tables, objects, and traps, the time consuming and resource intensive task of continuously polling each row in the entire atmVplTable and atmVclTable can be avoided.

トラップは永久VPLsにとのVCLの状態の変化を検出するために定義されています。各永久VPLまたはVCLの現在のアップ/ダウン状態は、それぞれ、atmVplOperStatus又はatmVclOperStatusオブジェクトで示されています。いくつかのテーブルとオブジェクトと1つのトラップは迅速にネットワーク管理者を支援し、効率的に永久仮想リンクの状態の変化を検出するために定義されています。これらのテーブル、オブジェクト、およびトラップの使用により、時間がかかり、全体atmVplTableとatmVclTableの各行を回避することができる連続ポーリングの集中的タスクをリソース。

The atmIntfPvcFailures counter and the atmIntfCurrentlyFailingPVpls and atmIntfCurrentlyFailingPVcls gauges provide a quick means of determining the status of all PVPLs and PVCLs on an interface. The atmCurrentlyFailingPVplTable and the atmCurrentlyFailingPVclTable list all of the problematic PVPLs and PVCLs, respectively, allowing them to be quickly identified.

atmIntfPvcFailuresカウンタとatmIntfCurrentlyFailingPVplsとatmIntfCurrentlyFailingPVclsゲージは、インターフェイス上のすべてのPVPLsとPVCLsの状態を決定するための迅速な手段を提供します。 atmCurrentlyFailingPVplTableとatmCurrentlyFailingPVclTableリストそれぞれ問題PVPLsとPVCLsのすべては、それらを迅速に識別することが可能になります。

The atmIntfPvcFailuresTrap is generated just after a PVPL or PVCL on a particular interface leaves the 'up' operational state. Managers can then determine which PVPLs and/or PVCLs are failing by reading the atmCurrentlyFailingPVplTable and the atmCurrentlyFailingPVclTable. Generation of the atmIntfPvcFailuresTrap is rate limited by suppressing all traps that would occur within atmIntfPvcNotificationInterval of a previous trap for the same interface. Managers should continuously poll the tables and objects mentioned above for at least this amount of time in order to keep up with the state of the network.

atmIntfPvcFailuresTrapは、特定のインターフェイス上のPVPLまたはPVCLが動作状態「まで」葉直後に生成されます。管理者は次にPVPLs及び/又はPVCLsはatmCurrentlyFailingPVplTableとatmCurrentlyFailingPVclTableを読み取ることによって、失敗しているかを決定することができます。 atmIntfPvcFailuresTrapの世代は、同じインターフェイスのために、前のトラップのatmIntfPvcNotificationInterval以内に起こるすべてのトラップを抑制することによって制限されています。管理者は、継続的にネットワークの状態に追いつくために時間の少なくともこの量のために、上記のテーブルとオブジェクトをポーリングする必要があります。

It is RECOMMENDED that implementers consider the security features as provided by the SNMPv3 framework (see [RFC3410], section 8), including full support for the SNMPv3 cryptographic mechanisms (for authentication and privacy).

実装がSNMPv3フレームワークで提供するようにセキュリティ機能を考えることが推奨される（認証とプライバシーのために）SNMPv3の暗号化メカニズムの完全なサポートを含む、（[RFC3410]セクション8を参照）。

Further, deployment of SNMP versions prior to SNMPv3 is NOT RECOMMENDED. Instead, it is RECOMMENDED to deploy SNMPv3 and to enable cryptographic security. It is then a customer/operator responsibility to ensure that the SNMP entity giving access to an instance of this MIB module is properly configured to give access to the objects only to those principals (users) that have legitimate rights to indeed GET or SET (change/create/delete) them.

さらに、SNMPv3の前のSNMPバージョンの展開はお勧めしません。代わりに、SNMPv3を展開すると、暗号化セキュリティを有効にすることをお勧めします。このMIBモジュールのインスタンスへのアクセスを与えるSNMP実体が適切にのみプリンシパル（ユーザ）にオブジェクトへのアクセスを提供するように設定されていることを確認するために、顧客/オペレータ責任実際にGETまたはSET（変化への正当な権利を有することです/）/削除、それらを作成します。
