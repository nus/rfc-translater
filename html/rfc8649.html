Hash Of Root Key Certificate Extension

ルートキー証明書拡張のハッシュ

Abstract

概要

This document specifies the Hash Of Root Key certificate extension. This certificate extension is carried in the self-signed certificate for a trust anchor, which is often called a Root Certification Authority (CA) certificate. This certificate extension unambiguously identifies the next public key that will be used at some point in the future as the next Root CA certificate, eventually replacing the current one.

このドキュメントでは、ルートキーのハッシュ証明書拡張機能を指定しています。 この証明書拡張機能は、ルートアンカー（CA）証明書と呼ばれることが多いトラストアンカーの自己署名証明書に含まれています。 この証明書拡張機能は、将来のある時点で次のルートCA証明書として使用される次の公開キーを明確に識別し、最終的に現在の証明書を置き換えます。

1. Introduction
1. はじめに

This document specifies the Hash Of Root Key X.509 version 3 certificate extension. The extension is an optional addition to the Internet X.509 Public Key Infrastructure Certificate and Certificate Revocation List (CRL) Profile [RFC5280]. The certificate extension facilitates the orderly transition from one Root Certification Authority (CA) public key to the next. It does so by publishing the hash value of the next-generation public key in the current self-signed certificate. This hash value is a commitment to a particular public key in the next-generation self-signed certificate. This commitment allows a relying party to unambiguously recognize the next-generation self-signed certificate when it becomes available, install the new self-signed certificate in the trust anchor store, and eventually remove the previous one from the trust anchor store.

このドキュメントでは、ルートキーのハッシュX.509バージョン3証明書拡張機能を指定しています。 この拡張は、インターネットX.509公開キー基盤の証明書および証明書失効リスト（CRL）プロファイル[RFC5280]へのオプションの追加です。 証明書の拡張機能により、あるルート証明機関（CA）の公開キーから次のキーへの規則的な移行が容易になります。 これは、現在の自己署名証明書に次世代の公開キーのハッシュ値を公開することにより行われます。 このハッシュ値は、次世代の自己署名証明書内の特定の公開キーに対するコミットメントです。 このコミットメントにより、証明書利用者は、利用可能になったときに次世代の自己署名証明書を明確に認識し、新しい自己署名証明書をトラストアンカーストアにインストールし、最終的にはトラストアンカーストアから以前の証明書を削除できます。

A Root CA certificate MAY include the Hash Of Root Key certificate extension to provide the hash value of the next public key that will be used by the Root CA.

ルートCA証明書には、ルートCAが使用する次の公開キーのハッシュ値を提供するために、ルートキーのハッシュ証明書拡張機能を含めることができます。

2. Overview
2. 概要

Before the initial deployment of the Root CA, the following are generated:

ルートCAの初期展開の前に、以下が生成されます。

R1 = The initial Root key pair R2 = The second-generation Root key pair H2 = Thumbprint (hash) of the public key of R2 C1 = Self-signed certificate for R1, which also contains H2

R1 =最初のルートキーペアR2 =第2世代のルートキーペアH2 = R2の公開キーのThumb印（ハッシュ）C1 = R1の自己署名証明書（H2も含む）

C1 is a self-signed certificate, and it contains H2 within the HashOfRootKey extension. C1 is distributed as part of the initial system deployment. The HashOfRootKey certificate extension is described in Section 3.

C1は自己署名証明書であり、HashOfRootKey拡張内にH2が含まれています。 C1は、最初のシステム展開の一部として配布されます。 HashOfRootKey証明書拡張機能については、セクション3で説明します。

When the time comes to replace the initial Root CA certificate, R1, the following are generated:

最初のルートCA証明書であるR1を置き換えるときが来たら、以下が生成されます。

R3 = The third-generation Root key pair H3 = Thumbprint (hash) the public key of R3 C2 = Self-signed certificate for R2, which contains H3

R3 =第三世代のルートキーペアH3 = Thumb印（ハッシュ）R3の公開キーC2 = H3を含むR2の自己署名証明書

This is an iterative process. That is, R4 and H4 are generated when it is time for C3 to replace C2, and so on.

これは反復プロセスです。 つまり、C3がC2を置き換えるときなどに、R4とH4が生成されます。

The successor to the Root CA self-signed certificate can be delivered by any means. Whenever a new Root CA self-signed certificate is received, the recipient is able to verify that the potential Root CA certificate links back to a previously authenticated Root CA certificate with the HashOfRootKey certificate extension. That is, the recipient verifies the signature on the self-signed certificate and verifies that the hash of the DER-encoded SubjectPublicKeyInfo from the potential Root CA certificate matches the value from the HashOfRootKey certificate extension of the current Root CA certificate. Checking the self-signed certificate signature ensures that the certificate contains the subject name, public key algorithm identifier, and public key algorithm parameters intended by the key owner; these are important inputs to certification path validation as defined in Section 6 of [RFC5280]. Checking the hash of the SubjectPublicKeyInfo ensures that the certificate contains the intended public key. If either check fails, then the potential Root CA certificate is not a valid replacement, and the recipient continues to use the current Root CA certificate. If both checks succeed, then the recipient adds the potential Root CA certificate to the trust anchor store. As discussed in Section 5, the recipient can remove the current Root CA certificate immediately in some situations. In other situations, the recipient waits an appropriate amount of time to ensure that existing certification paths continue to validate.

ルートCAの自己署名証明書の後継は、任意の方法で配信できます。新しいルートCA自己署名証明書が受信されるたびに、受信者は、潜在的なルートCA証明書が、HashOfRootKey証明書拡張子を持つ以前に認証されたルートCA証明書にリンクしていることを確認できます。つまり、受信者は自己署名証明書の署名を検証し、潜在的なルートCA証明書のDERエンコードされたSubjectPublicKeyInfoのハッシュが現在のルートCA証明書のHashOfRootKey証明書拡張の値と一致することを検証します。自己署名証明書の署名を確認すると、証明書にサブジェクト名、公開鍵アルゴリズム識別子、および鍵所有者が意図した公開鍵アルゴリズムのパラメーターが含まれていることが保証されます。これらは、[RFC5280]のセクション6で定義されている認証パス検証への重要な入力です。 SubjectPublicKeyInfoのハッシュをチェックすると、証明書に目的の公開キーが含まれていることが保証されます。いずれかのチェックが失敗した場合、潜在的なルートCA証明書は有効な代替ではなく、受信者は現在のルートCA証明書を引き続き使用します。両方のチェックが成功した場合、受信者は潜在的なルートCA証明書をトラストアンカーストアに追加します。セクション5で説明したように、受信者は状況によっては現在のルートCA証明書をすぐに削除できます。他の状況では、受信者は適切な時間だけ待機して、既存の証明書パスが引き続き検証されるようにします。

3. Hash Of Root Key Certificate Extension
3. ルートキー証明書拡張のハッシュ

The HashOfRootKey certificate extension MUST NOT be critical.

HashOfRootKey証明書の拡張は重要ではありません。

The following ASN.1 [X680] [X690] syntax defines the HashOfRootKey certificate extension:

次のASN.1 [X680] [X690]構文は、HashOfRootKey証明書拡張を定義します。 

   ext-HashOfRootKey EXTENSION ::= {    -- Only in Root CA certificates
      SYNTAX         HashedRootKey
      IDENTIFIED BY  id-ce-hashOfRootKey
      CRITICALITY    {FALSE} }
        
   HashedRootKey ::= SEQUENCE {
      hashAlg        HashAlgorithm,        -- Hash algorithm used
      hashValue      OCTET STRING }        -- Hash of DER-encoded
                                           --   SubjectPublicKeyInfo
        
   id-ce-hashOfRootKey  ::=  OBJECT IDENTIFIER { 1 3 6 1 4 1 51483 2 1 }

The definitions of EXTENSION and HashAlgorithm can be found in [RFC5912].

EXTENSIONとHashAlgorithmの定義は[RFC5912]にあります。

The hashAlg indicates the one-way hash algorithm that was used to compute the hash value.

hashAlgは、ハッシュ値の計算に使用された一方向ハッシュアルゴリズムを示します。

The hashValue contains the hash value computed from the next-generation public key. The public key is the DER-encoded SubjectPublicKeyInfo as defined in [RFC5280].

hashValueには、次世代の公開キーから計算されたハッシュ値が含まれます。 公開鍵は、[RFC5280]で定義されているDERエンコードされたSubjectPublicKeyInfoです。

5. Operational Considerations
5. 運用上の考慮事項

Guidance on the transition from one root key to another is available in Section 4.4 of [RFC4210]. Of course, a root key is also known as a trust anchor. In particular, the oldWithNew and newWithOld advice ensures that relying parties are able to validate certificates issued under the current Root CA certificate and the next-generation Root CA certificate throughout the transition. The notAfter field in the oldWithNew certificate MUST cover the validity period of all unexpired certificates issued under the old Root CA private key. Further, this advice SHOULD be followed by Root CAs to avoid the need for all relying parties to make the transition at the same time.

1つのルートキーから別のルートキーへの移行に関するガイダンスは、[RFC4210]のセクション4.4で利用できます。 もちろん、ルートキーはトラストアンカーとも呼ばれます。 特に、oldWithNewおよびnewWithOldのアドバイスにより、信頼関係者は、移行中に現在のルートCA証明書および次世代ルートCA証明書の下で発行された証明書を検証できます。 oldWithNew証明書のnotAfterフィールドは、古いルートCA秘密鍵の下で発行されたすべての期限切れのない証明書の有効期間をカバーしなければなりません。 さらに、すべての証明書利用者が同時に移行を行う必要性を回避するために、ルートCAがこのアドバイスに従う必要があります。

After issuing the newWithOld certificate, the Root CA MUST stop using the old private key to sign certificates.

newWithOld証明書を発行した後、ルートCAは証明書に署名するために古い秘密キーの使用を停止する必要があります。

Some enterprise and application-specific environments offer a directory service or certificate repository to make certificate and CRLs available to relying parties. Section 3 in [RFC5280] describes a certificate repository. When a certificate repository is available, the oldWithNew and newWithOld certificates SHOULD be published before the successor to the current Root CA self-signed certificate is released. Recipients that are able to obtain the oldWithNew certificate SHOULD immediately remove the old Root CA self-signed certificate from the trust anchor store.

一部のエンタープライズおよびアプリケーション固有の環境では、証明書およびCRLを証明書利用者が利用できるようにするディレクトリサービスまたは証明書リポジトリを提供しています。 [RFC5280]のセクション3では、証明書リポジトリについて説明しています。 証明書リポジトリが利用可能な場合、oldWithNewおよびnewWithOld証明書は、現在のルートCA自己署名証明書の後継がリリースされる前に発行される必要があります。 oldWithNew証明書を取得できる受信者は、トラストアンカーストアから古いルートCA自己署名証明書をすぐに削除する必要があります。

In environments without such a directory service or repository, like the Web PKI, recipients need a way to obtain the oldWithNew and newWithOld certificates. The Root CA SHOULD include the subject information access extension [RFC5280] with the accessMethod set to id-ad-caRepository and the assessLocation set to the HTTP URL that can be used to fetch a DER-encoded "certs-only" (simple PKI response) message as specified in [RFC5272] in all of their self-signed certificates. The Root CA SHOULD publish the "certs-only" message with the oldWithNew certificate and the newWithOld certificate before the subsequent Root CA self-signed certificate is released. The "certs-only" message format allows certificates to be added and removed from the bag of certificates over time, so the same HTTP URL can be used throughout the lifetime of the Root CA.

Web PKIなど、このようなディレクトリサービスまたはリポジトリがない環境では、受信者はoldWithNewおよびnewWithOld証明書を取得する方法が必要です。 ルートCAは、accessMethodがid-ad-caRepositoryに設定され、amesLocationがDERエンコードされた「証明書のみ」（単純なPKI応答を取得するために使用できるHTTP URLに設定されたサブジェクト情報アクセス拡張[RFC5280]を含む必要があります ）すべての自己署名証明書の[RFC5272]で指定されているメッセージ。 ルートCAは、後続のルートCA自己署名証明書がリリースされる前に、oldWithNew証明書とnewWithOld証明書を使用して「証明書のみ」メッセージを発行する必要があります。 「証明書のみ」のメッセージ形式では、証明書を時間の経過とともに証明書のバッグに追加したり、証明書のバッグから削除したりできるため、ルートCAの存続期間を通じて同じHTTP URLを使用できます。

In environments without such a directory service or repository, recipients SHOULD keep both the old and replacement Root CA self-signed certificates in the trust anchor store for some amount of time to ensure that all end-entity certificates can be validated until they expire. The recipient MAY keep the old Root CA self-signed certificate until all of the certificates in the local cache that are subordinate to it have expired.

そのようなディレクトリサービスまたはリポジトリのない環境では、受信者は、古いCAと代替の両方のルートCA自己署名証明書を一定期間トラストアンカーストアに保持して、すべてのエンドエンティティ証明書が期限切れになるまで検証できるようにする必要があります（SHOULD）。 受信者は、その下位にあるローカルキャッシュ内のすべての証明書の有効期限が切れるまで、古いルートCA自己署名証明書を保持することができます。

Certification path construction is more complex when the trust anchor store contains multiple self-signed certificates with the same distinguished name. For this reason, the replacement Root CA self-signed certificate SHOULD contain a different distinguished name than the one it is replacing. One approach is to include a number as part of the name that is incremented with each generation, such as "Example CA", "Example CA G2", "Example CA G3", and so on.

トラストアンカーストアに同じ識別名を持つ複数の自己署名証明書が含まれる場合、証明書パスの構築はより複雑になります。 このため、置換ルートCAの自己署名証明書には、置換するものとは異なる識別名が含まれている必要があります。 1つのアプローチは、「サンプルCA」、「サンプルCA G2」、「サンプルCA G3」などのように、世代ごとに増分される名前の一部として番号を含めることです。

Changing names from one generation to another can lead to confusion when reviewing the history of a trust anchor store. To assist with such review, a recipient MAY create an audit entry to capture the old and replacement self-signed certificates.

ある世代から別の世代に名前を変更すると、トラストアンカーストアの履歴を確認するときに混乱が生じる可能性があります。 そのようなレビューを支援するために、受信者は監査エントリを作成して、古い自己署名証明書と置換自己署名証明書をキャプチャすることができます。

The Root CA must securely back up the yet-to-be-deployed key pair. If the Root CA stores the key pair in a hardware security module and that module fails, the Root CA remains committed to the key pair that is no longer available. This leaves the Root CA with no alternative but to deploy a new self-signed certificate that contains a newly generated key pair in the same manner as the initial self-signed certificate, thus losing the benefits of the Hash Of Root Key certificate extension altogether.

ルートCAは、まだ展開されていないキーペアを安全にバックアップする必要があります。 ルートCAがハードウェアセキュリティモジュールにキーペアを保存し、そのモジュールに障害が発生した場合、ルートCAは使用できなくなったキーペアにコミットされたままになります。 これにより、ルートCAには代替手段がなくなりますが、最初の自己署名証明書と同じ方法で新しく生成されたキーペアを含む新しい自己署名証明書を展開することになるため、ルートキーのハッシュ証明書拡張機能のメリットを完全に失います。

6. Security Considerations
6. セキュリティに関する考慮事項

The security considerations from [RFC5280] apply, especially the discussion of self-issued certificates.

[RFC5280]のセキュリティに関する考慮事項、特に自己発行証明書の議論が適用されます。

The Hash Of Root Key certificate extension facilitates the orderly transition from one Root CA public key to the next by publishing the hash value of the next-generation public key in the current certificate. This allows a relying party to unambiguously recognize the next-generation public key when it becomes available; however, the full public key is not disclosed until the Root CA releases the next-generation certificate. In this way, attackers cannot begin to analyze the public key before the next-generation Root CA self-signed certificate is released.

Hash Of Root Key証明書拡張機能は、現在の証明書で次世代公開キーのハッシュ値を公開することにより、あるルートCA公開キーから次の公開キーへの規則正しい移行を促進します。 これにより、証明書利用者は次世代の公開キーが利用可能になったときにそれを明確に認識することができます。 ただし、ルートCAが次世代の証明書をリリースするまで、完全な公開キーは公開されません。 この方法では、次世代のルートCA自己署名証明書がリリースされる前に、攻撃者は公開鍵の分析を開始できません。

The Root CA needs to ensure that the public key in the next-generation certificate is as strong or stronger than the key that it is replacing. Of course, a significant advance in cryptoanalytic capability can break the yet-to-be-deployed key pair. Such advances are rare and difficult to predict. If such an advance occurs, the Root CA remains committed to the now broken key. This leaves the Root CA with no alternative but to deploy a new self-signed certificate that contains a newly generated key pair, most likely using a different signature algorithm, in the same manner as the initial self-signed certificate, thus losing the benefits of the Hash Of Root Key certificate extension altogether.

ルートCAは、次世代証明書の公開キーが、置換するキーと同じかそれよりも強力であることを確認する必要があります。 もちろん、暗号分析機能の大幅な進歩により、まだ展開されていないキーペアが破損する可能性があります。 このような進歩はまれであり、予測が困難です。 そのような前進が発生した場合、ルートCAは現在破損しているキーにコミットされたままになります。 これにより、ルートCAには代替手段がなくなりますが、最初の自己署名証明書と同じ方法で、おそらく異なる署名アルゴリズムを使用して、新たに生成されたキーペアを含む新しい自己署名証明書を展開するため、 Hash Of Root Key証明書拡張機能全体。

The Root CA needs to employ a hash function that is resistant to preimage attacks [RFC4270]. A first-preimage attack against the hash function would allow an attacker to find another input that results in the hash value of the next-generation public key that was published in the current certificate. For the attack to be successful, the input would have to be a valid SubjectPublicKeyInfo that contains a public key that corresponds to a private key known to the attacker. A second-preimage attack becomes possible once the Root CA releases the next-generation public key, which makes the input to the hash function available to the attacker and everyone else. Again, the attacker needs to find a valid SubjectPublicKeyInfo that contains the public key that corresponds to a private key known to the attacker. If the employed hash function is broken after the Root CA publishes the self-signed certificate with the HashOfRootKey certificate extension, an attacker would be able to trick the recipient into installing the incorrect next-generation certificate in the trust anchor store.

ルートCAは、プリイメージ攻撃に耐性のあるハッシュ関数を使用する必要があります[RFC4270]。ハッシュ関数に対する最初のプリイメージ攻撃により、攻撃者は、現在の証明書で公開された次世代公開キーのハッシュ値をもたらす別の入力を見つけることができます。攻撃を成功させるには、攻撃者が知っている秘密キーに対応する公開キーを含む有効なSubjectPublicKeyInfoを入力する必要があります。ルートCAが次世代の公開キーをリリースすると、2番目のプリイメージ攻撃が可能になります。これにより、ハッシュ関数への入力が攻撃者および他のすべてのユーザーに利用可能になります。繰り返しますが、攻撃者は、攻撃者が知っている秘密鍵に対応する公開鍵を含む有効なSubjectPublicKeyInfoを見つける必要があります。ルートCAがHashOfRootKey証明書拡張機能を使用して自己署名証明書を公開した後に、採用されたハッシュ関数が破損した場合、攻撃者は受信者をだまして不正な次世代証明書をトラストアンカーストアにインストールさせることができます。

If an early release of the next-generation public key occurs and the Root CA is concerned that attackers were given too much lead time to analyze that public key, then the Root CA can transition to a freshly generated key pair by rapidly performing two transitions. After the first transition, the Root CA is using the key pair that suffered the early release, and that transition causes the Root CA to generate the subsequent Root key pair. The second transition occurs when the Root CA is confident that the population of relying parties has completed the first transition, and it takes the Root CA to the freshly generated key pair. Of course, the second transition also causes the Root CA to generate another key pair that is reserved for future use. Queries for the CRLs associated with certificates that are subordinate to the self-signed certificate can give some indication of the number of relying parties that are still actively using the self-signed certificates.

次世代の公開キーの早期リリースが発生し、ルートCAが攻撃者にその公開キーを分析するのに時間がかかりすぎることを懸念している場合、ルートCAは2つの移行を迅速に実行することにより、新たに生成されたキーペアに移行できます。 最初の移行後、ルートCAは初期リリースの影響を受けたキーペアを使用し、その移行によりルートCAが後続のルートキーペアを生成します。 2回目の移行は、ルートCAが証明書利用者の集団が最初の移行を完了したと確信したときに発生し、新たに生成されたキーペアにルートCAを取得します。 もちろん、2番目の移行により、ルートCAは将来の使用のために予約されている別のキーペアを生成します。 自己署名証明書に従属する証明書に関連付けられているCRLのクエリは、自己署名証明書をまだアクティブに使用している証明書利用者の数を示すことができます。
